# AIエージェント勉強会 トークメモ（60分 / 実運用ベース）

## 0. このメモの方針

- **時間: 60分**
- **背骨は「概念の正確な網羅」ではなく「私が実際にどう使っているか」**。
  自分の実プロジェクト（このOIDC/OAuth Provider OSS）での運用を主役に語る。
- 概念は「私が使っているもの／使っていないもの」を説明する**ための道具**として、その都度かみ砕いて触れる。
- **キモは "使っていないものの理由"**。なぜ採用していないかを自分の言葉で語れると、聞き手にとって一番価値が出る。
- 話者前提: Copilot経験あり／Claude Code はゴリゴリ未経験／最新動向は未追従。聞き手も同程度を想定。

> ⚠️ 下の「使っている／使っていない」分類は、このリポジトリの設定と過去のやり取りから推定したもの。
> **当日までに自分の実感と照合して微修正すること**（特に MCP/Serena と Hooks の使用実態）。

## 2. つかみ（5分）

- Copilotの世界観: 人間が書く、AIが隣で次の行を提案。**主役は人間**。
- エージェントの世界観: 結果を言う、AIが探索→計画→編集→テスト→修正。**人間は監督**。
- 今日の宣言: 「最新機能を全部正確に説明する会ではない。**私が実際にこのOSSをどう回しているか**、使っているもの・使っていないものとその理由を話す。だから偏りもあるし、それでいい」
- 補足: 「Copilotにも今はagentモードがある。別物ではなく延長線上」と言うと聞き手が地続きで入れる。

## 3. 私のセットアップ全体像（8分）

- **画面で `CLAUDE.md` を開いて見せる**。
- 語り: 「これがAIへの“就業規則”。毎回自動で読まれる。中身はコンセプト → ツール役割分担 → 7フェーズの開発ワークフロー → 実装ルール → テスト規約」
- 7フェーズ（調査→設計→設計協議→テスト先行→コードレビュー→セルフレビュー→完了レポート）を1枚で見せる。
- ここで言いたいこと: 「**私はAIに“自由にやらせる”のではなく、自分の手順を強制している**。これが私のスタイルの根っこ」

## 4. 実際の1タスクの流れ（18分）＝ 使っているものの説明

実タスク（例: 直近のP0コンプライアンス修正）を題材に、流れの中で“使っているもの”を紹介する。

1. **タスクファイル（`tasks/p0-*.md`, P0〜P3）を見せる**
   - 概念: AIに優先度と要件を渡す“作業指示書”。
   - なぜ使う: 優先度を共有し、TDDと仕様参照を強制できるから。
2. **調査フェーズ → Skill `/tech-research`（Gemini CLI）**
   - 概念: Skill＝再利用する手順書。`/名前`で呼ぶ。
   - なぜ使う: 仕様調査をGeminiに外出しして**Claudeの文脈を節約**するため。
3. **設計協議 → Skill `/design-discussion`（Codex CLI）**
   - 概念: 外部の別AIにセカンドオピニオンを取る分業。
   - なぜ使う: Claude単体で完結させず、レビュー役を別系統に持たせ品質を担保するため（＝マルチツール・オーケストレーション。考え方は orchestrator-worker で現代的）。
4. **テスト先行（TDD）→ 単発スコープ指示で実装**
   - なぜこの粒度: リミット対策（→ パート6で核心を語る）。
5. **完了したら Hooks でビープ通知**
   - 概念: ライフサイクルの節目で走るスクリプト。
   - なぜ使う: 「長い作業の間に席を立てる。**エージェントの価値は“待てること”**。終わったら音で呼んでもらう」← 体験談として強い。
6. （触れる程度）**MCP / Serena**: コードの意味解析をAIに効かせる接続。「設定して常時効かせている」レベルで紹介。

> このパートのメッセージ: 「使っているのは CLAUDE.md・Skills・外部CLI分業・タスクファイル・Hooks・MCP。**派手な新機能ではなく、地味な“手順の固定化”が中心**」

## 5. 使っていないもの と その理由（14分）＝ 本日の主役

「使っていない」を堂々と語るのがこの勉強会の差別化。各項目「概念を一言 → なぜ私は使わないか」。

| 機能 | 概念（一言） | なぜ私は使っていないか |
|---|---|---|
| **Plan Mode** | 読み取り専用で調査し実装計画を出して承認待ち | 2回ほど使うとすぐ使用量リミットに到達した。探索＋計画が文脈に居座り毎ターン再課金される。私は自分で細かく舵を取るので“防ぐべき手戻り”がそもそも少なく、固定オーバーヘッドだけ損になる（詳細はパート6） |
| **ネイティブ Subagents**（`.claude/agents/`） | 別文脈で動く専用エージェント、要約だけ戻る | 先に **外部CLI（Codex/Gemini）で分業体制を組んでしまった**ので役割は既に満たせている。ネイティブに寄せれば再現性は上がるが、移行コストとのトレードオフで今は据え置き |
| **Agent Teams** | 複数のClaude Codeを並列、リーダーが統括 | ソロ開発で並列に割れる独立作業が少ない。得られるのは時間短縮で、**トークンはむしろ増える**。リミットが有限な自分には割が合わない |
| **Plugins** | Skills/Hooks等をまとめて配布・ワンクリック導入 | 一人開発で配布・共有の相手がいない。Skillsをローカルに置けば足りるので、パッケージ化の必要がまだ無い |

語りの締め: 「使わない理由はだいたい3つに集約される——**①リミット（コスト）②ソロ運用で不要 ③既存の自前体制で代替できている**。新機能=使うべき、ではない」

## 6. なぜ私は単発指示なのか：トークン/リミットの現実（10分）

“使わない理由”の核心。実体験なので一番熱を込めて話せる所。

- **2つの別物を分ける**: 「1タスクの総トークン削減」と「使用量リミット（ローリング枠）への到達速度」は別の指標。
- **Plan Mode は後者に逆効果**: 計画作成のため大量探索＋思考を前倒し → その探索結果と計画が文脈に居座る → 毎ターン文脈丸ごと再課金 → 枠が急速に埋まる。
- **私の結論**: 「スコープを小さく切った単発指示の方が、リミットまで長く作業できた」。
- 使い分けの一般化（聞き手の持ち帰り）:

| 状況 | 私の選択 |
|---|---|
| 小さい・スコープ明確・自分で逐次修正 | 単発指示 |
| 多ファイル横断・方針不確実・外すと手戻り大 | （例外的に）Plan Mode |
| リミットが近い／長く続けたい | 単発指示一択 |

- ついでに渡す節約Tips: ①スコープを小さく ②重い処理だけ別文脈に隔離 ③CLAUDE.mdは短く（毎ターン再送）④無関係タスク切替で `/clear`。

## 7. まとめ＋Q&A（5分）

- 持ち帰り3点:
  1. AIエージェントは「指揮するもの」。**自分の手順を固定して渡す**のが効く。
  2. 新機能は“使うべき”ではない。**自分の制約（コスト/ソロ/既存体制）で取捨**する。
  3. Plan Mode等は万能ではない。**リミットという現実**で判断が変わる。
- 想定Q&A:
  - Q: Copilotから乗り換える? → 用途で併用。補完はCopilot、まとまった作業はエージェント。
  - Q: トークン高くない? → 高くなりうる。だから単発指示中心にしている。
  - Q: 業務で安全? → 計画/差分レビュー/テスト検証のループを守れば監督可能。丸投げ禁物。
  - Q: 何から始める? → CLAUDE.mdを1枚書く → 小タスクを単発指示 → 必要時だけ機能を足す。
