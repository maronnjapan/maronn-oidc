# タスク05・06 仕様確認調査結果

## タスク05: 認可コードTTL

### 調査1: OAuth 2.1 認可コードの有効期間
- OAuth 2.1では最大10分を推奨。より短い期間（60秒）が望ましい
- 5分（300秒）は許容範囲内 ✅

### 調査2: 有効期限チェックの境界条件
- RFC 7519 (JWT) では `now >= exp` のとき期限切れ（`<=` が正しい）
- `expiresAt < now` は `now == expiresAt` の瞬間を「有効」と誤判定する
- 仕様に厳密に従うなら `expiresAt <= now` が正しい
- 現実の実装（`< now`）は1秒未満の差であり実害はほぼない

### 調査3: used フラグ + TTLの組み合わせ
- 両方必須: ワンタイム制御はリプレイ攻撃防止、TTLは漏洩リスク時間制限
- 現在の実装は両方実装済み ✅

## タスク06: RSA鍵プロバイダー

### 調査4: OIDC Core 1.0 Section 10.1 署名鍵要件
- 仕様上、鍵の生成タイミングに明確な規定なし
- ただしエフェメラル鍵（再起動ごとに変わる鍵）はRPキャッシュとの不整合でエラーを招く
- 実運用では永続的な鍵 + 計画的ローテーションが推奨

### 調査5: RFC 7517 kid (Key ID) クレーム
- RFC 7517自体ではOPTIONAL
- JWKSで複数鍵を公開する場合、IDトークンのJWSヘッダーとのkid照合が必要
- 実質的にJWKS運用では必須
