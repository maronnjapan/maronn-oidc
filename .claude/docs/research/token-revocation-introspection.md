## OAuth 2.0 トークン無効化 (RFC 7009) およびトークンイントロスペクション (RFC 7662) に関する調査レポート

### 概要
本レポートは、OAuth 2.0のトークン無効化 (RFC 7009) およびトークンイントロスペクション (RFC 7662) の仕様、ならびにOAuth 2.1およびIETF Security BCPがこれらの機能に与える影響について詳細に調査したものである。TypeScript OIDCプロバイダライブラリの実装におけるベストプラクティスと採用すべき方針を提示することを目的とする。

---

### 1. RFC 7009 無効化エンドポイントの仕様: HTTPメソッド、パラメータ、成功ステータスコード、エラーレスポンス、Cache-Control/Pragmaヘッダ

#### 仕様の規定
RFC 7009 Section 2.1 にて、無効化エンドポイントは `POST` メソッドを使用し、`application/x-www-form-urlencoded` 形式でパラメータを受け取る。必須パラメータは無効化対象のトークンを示す `token` であり、オプションでトークンの種類を示す `token_type_hint` を指定できる。クライアント認証は必須である。成功レスポンスは空のボディを持つ `200 OK` であり、トークンが存在しない場合でも `200 OK` を返すことでサイドチャネル攻撃を防ぐ。エラーレスポンスは OAuth 2.0 (RFC 6749 Section 5.2) に従い `400 Bad Request` とエラーボディを返す。レスポンスには `Cache-Control: no-store` および `Pragma: no-cache` ヘッダを含め、キャッシュを禁止しなければならない (`MUST NOT`)。

#### ベストプラクティス
無効化エンドポイントは常に `POST` メソッドで実装し、クライアント認証を厳格に適用する。特に機密クライアントには `client_secret_basic` または `client_secret_post` を必須とする。トークンが存在しない場合でも `200 OK` を返すことで、トークンの存在を外部から推測させないようにする。エラー時には OAuth 2.0 の標準エラー形式に従い、詳細なエラー情報はデバッグ時のみに限定し、本番環境では抽象的なエラーに留める。レスポンスにはキャッシュ禁止ヘッダを必ず含める。

#### 採用する方針の選択肢
*   **RFC 7009 厳格準拠:** `POST` メソッド、`token` パラメータ必須、`token_type_hint` 任意、クライアント認証必須、成功時 `200 OK`（空ボディ）、エラー時 `400 Bad Request`（OAuth 2.0 エラー形式）、キャッシュ禁止ヘッダを実装する。
*   **トークン非存在時の挙動:** RFC 7009 Section 2.2 の推奨に従い、トークンが存在しない場合でも `200 OK` を返すことで、トークンの存在を外部から推測させないようにする。
*   **エラー情報の粒度:** 本番環境ではセキュリティを優先し、RFC 準拠の抽象的なエラーレスポンスに限定する。開発・テスト環境向けに、より詳細なエラー情報を返すオプションを検討する。

### 2. `token_type_hint`: REQUIRED/OPTIONAL、値が不明な場合の挙動、`access_token` / `refresh_token` 以外の値

#### 仕様の規定
RFC 7009 Section 2.1 によると、`token_type_hint` パラメータは `OPTIONAL` である。認可サーバーは、このヒントを使用してトークンを効率的に特定できる。ヒントが提供されない場合、認可サーバーは利用可能なすべてのトークンタイプ（例：アクセス、リフレッシュ）に対してトークンを検索しなければならない。ヒントが提供され、それが認可サーバーによって認識されない場合、認可サーバーは `400 Bad Request` と `unsupported_token_type` エラーを返してもよい (`MAY`)。しかし、トークンが有効な場合は、ヒントを無視してトークンを無効化し、`200 OK` を返すこともできる。

#### ベストプラクティス
クライアントは可能な限り `token_type_hint` を提供すべきであり、これにより認可サーバーの処理効率が向上する。認可サーバーは、ヒントが提供されない場合でも、すべてのトークンタイプを検索してトークンを無効化できるように実装すべきである。セキュリティ上の理由（情報漏洩防止）から、認識されない `token_type_hint` に対して `unsupported_token_type` エラーを返すよりも、ヒントを無視してトークンを無効化（またはトークンが存在しない場合は `200 OK`）する方が望ましい。

#### 採用する方針の選択肢
*   **柔軟なヒント処理:** `token_type_hint` が提供されない場合は、アクセス・リフレッシュトークンの両方を検索する。ヒントが `access_token` または `refresh_token` 以外の場合でも、ヒントを無視して両方のタイプを検索し、トークンが見つかれば無効化して `200 OK` を返す。これにより、クライアントが誤ったヒントを送っても処理を継続できる。
*   **将来的な拡張性:** `id_token` や `device_code` など、将来的に他のトークンタイプをサポートする可能性を考慮し、`token_type_hint` の拡張性を確保する。ただし、RFC 7009 は `access_token` と `refresh_token` のみを想定しているため、これら以外のタイプを無効化する際は、そのトークンのライフサイクルとセキュリティへの影響を慎重に評価する。

### 3. 無効化エンドポイントのクライアント認証: 機密クライアント vs 公開クライアントの要件

#### 仕様の規定
RFC 7009 Section 2.1 は、認可サーバーがリクエストを行うクライアントを認証しなければならない (`MUST`) と規定している。クライアント認証は OAuth 2.0 (RFC 6749 Section 2.3) で定義されている方法に従う。機密クライアントは `client_id` と `client_secret` を使用して認証されるべきであり、通常は `client_secret_basic` または `client_secret_post` が用いられる。公開クライアントは `client_secret` を持たないため、`client_id` のみで識別される。

#### ベストプラクティス
機密クライアントに対しては、`client_secret_basic` または `client_secret_post` による認証を必須とすることで、無効化リクエストの正当性を保証し、悪意のある第三者によるトークン無効化を防ぐ。公開クライアントに対しては `client_id` のみでリクエストを受け入れるが、公開クライアントが発行したトークンは機密クライアントのトークンよりもセキュリティリスクが低いと見なされることが多い。クライアント認証に失敗した場合は、`401 Unauthorized` または `400 Bad Request` を返し、OAuth 2.0 のエラーレスポンス形式に従う。

#### 採用する方針の選択肢
*   **RFC 準拠の必須認証:** 機密クライアントには `client_secret_basic` または `client_secret_post` を必須とし、公開クライアントには `client_id` のみを要求する。これは標準的なアプローチであり、このプロジェクトのターゲットである機密クライアントの要件にも合致する。
*   **公開クライアントの制限:** 公開クライアントからのトークン無効化リクエストを、特定の条件（例：発行されたトークンが PKCE を使用している場合のみ許可）で制限することを検討する。ただし、RFC 7009 自体は公開クライアントの無効化を制限していない。
*   **追加の認証メカニズム:** 将来的に、クライアント証明書（mTLS）などのより強力な認証メカニズムをサポートする可能性を考慮する。これは OAuth 2.1 や IETF Security BCP で推奨されるセキュリティ強化策である。

### 4. リフレッシュトークンが無効化された場合、関連するアクセストークンも無効化されるべきか？

#### 仕様の規定
RFC 7009 Section 2.1 には、「If the token passed to the revocation endpoint is a refresh token, the authorization server `SHOULD` also invalidate all access tokens that were obtained using that refresh token.」と明記されている。つまり、リフレッシュトークンが無効化された場合、認可サーバーは、そのリフレッシュトークンを使用して取得されたすべてのアクセストークンを無効化すべきである (`SHOULD`)。

#### ベストプラクティス
リフレッシュトークンが無効化された場合、関連するすべてのアクセストークンを無効化することは、セキュリティ上のベストプラクティスである。これにより、リフレッシュトークンが侵害された場合に、そのリフレッシュトークンから派生したアクセストークンも無効化され、攻撃者が既存のアクセストークンを悪用するのを防ぐ。リフレッシュトークンは長期的な認証情報であり、その侵害がより広範な影響を持つ可能性があるため、この措置は特に重要である。実装においては、リフレッシュトークンとアクセストークンの関連付けを永続ストアで管理し、リフレッシュトークン無効化時にその関連付けを解除または無効化するメカニズムが必要となる。

#### 採用する方針の選択肢
*   **「SHOULD」の厳格な遵守:** リフレッシュトークンが無効化された場合、そのリフレッシュトークンから発行されたすべてのアクセストークンを必ず無効化する。これはセキュリティを最大化するための推奨されるアプローチであり、このプロジェクトの OIDC プロバイダとして採用すべきである。
*   **グラント ID による管理:** このプロジェクトは既に `grantId` を使用して認可コードの再利用時に兄弟トークンを無効化しているため、この `grantId` をリフレッシュトークンとアクセストークンの関連付けにも利用し、リフレッシュトークン無効化時に同じ `grantId` を持つすべてのトークンを無効化する方針を採用する。
*   **JWT アクセストークンの扱い:** JWT アクセストークンはステートレスであるため、即座に無効化することはできない。これについては、後述の JWT に関する質問で詳細に検討する。ここでは、Opaque アクセストークンに焦点を当てる。

### 5. アクセストークンが無効化された場合、関連するリフレッシュトークンはどうなるか？

#### 仕様の規定
RFC 7009 Section 2.1 には、「If the token passed to the revocation endpoint is an access token, the authorization server `MAY` also invalidate the refresh token that was used to obtain the access token.」と記載されている。つまり、アクセストークンが無効化された場合、認可サーバーは、そのアクセストークンの取得に使用されたリフレッシュトークンも無効化してもよい (`MAY`)。

#### ベストプラクティス
アクセストークンの無効化時にリフレッシュトークンも無効化するかどうかは、ユースケースとセキュリティ要件に依存する。一般的には、アクセストークンは短命であり、リフレッシュトークンは長期的なセッションを維持するために使用される。アクセストークンの無効化が、必ずしもユーザーセッション全体の終了を意味するわけではないため、リフレッシュトークンを無効化しない選択肢も有効である。しかし、ユーザーが明示的にログアウトした場合など、セッション全体を終了させたい場合は、アクセストークンとリフレッシュトークンの両方を無効化することが望ましい。

#### 採用する方針の選択肢
*   **「MAY」の柔軟な解釈:** アクセストークンの無効化時にリフレッシュトークンを無効化しないことをデフォルトの挙動とする。これは、ユーザーが一時的にアクセストークンを無効化したいが、セッション自体は維持したい場合に有用である。
*   **明示的なセッション終了:** クライアントが明示的にセッションを終了させたい場合（例：ログアウト機能）、リフレッシュトークンも無効化するオプションを提供する。これは、無効化エンドポイントへのリクエストに特別なパラメータを追加するか、別のエンドポイントを用意することで実現できる。
*   **グラント ID による関連付け:** このプロジェクトは `grantId` を使用しているため、アクセストークンの無効化時に、そのアクセストークンが属する `grantId` に関連付けられたリフレッシュトークンを無効化するかどうかを制御するロジックを実装する。デフォルトでは無効化しないが、特定の条件（例：クライアントからの明示的な要求）で無効化する。

### 6. JWT アクセストークンと無効化: ステートレス性 vs イントロスペクション vs デナリスト、RFCs / BCP が示唆するトレードオフと緩和策

#### 仕様の規定
JWT は自己完結型であり、発行後に認可サーバーがその状態を管理する必要がないため、ステートレスである。これはスケーラビリティの利点をもたらすが、即時無効化を困難にする (RFC 7662 Section 4)。JWT の有効性を確認する手段として、認可サーバーのイントロスペクションエンドポイントを使用できる (RFC 7662)。JWT を即時無効化する一般的な方法として、無効化された JWT の識別子（JTI）をデナリスト（ブラックリスト）に登録する方法がある。リソースサーバーは、受け取った JWT の JTI がデナリストに存在しないことを確認する必要がある。IETF Security BCP (draft-ietf-oauth-security-topics Section 2.2.2) は、JWT アクセストークンの即時無効化は困難であり、デナリストはスケーラビリティの問題を引き起こす可能性があると指摘している。短い有効期限の JWT を使用し、リフレッシュトークンを介して新しい JWT を頻繁に発行することを推奨している。また、送信者制約付きトークン（Sender-Constrained Tokens）の使用を推奨している。

#### ベストプラクティス
JWT アクセストークンの有効期限を短く設定し、リフレッシュトークンを介して頻繁に更新することで、侵害された JWT の悪用期間を最小限に抑える。デナリストは、スケーラビリティとパフォーマンスの課題があるため、大規模なシステムでは慎重に検討する必要がある。デナリストを使用する場合は、高速な検索が可能な分散キャッシュシステムなどを利用する。リソースサーバーは、JWT の署名検証だけでなく、有効期限、発行者、対象オーディエンスなどの標準的な検証も行うべきである。機密性の高い操作を行うリソースサーバーでは、JWT のイントロスペクションを強制することで、認可サーバーによる即時無効化の恩恵を受けることができる。送信者制約付きトークン（例：mTLS クライアント証明書バインディング）を導入することで、JWT が盗まれた場合でも、正当なクライアント以外が使用することを困難にする。

#### 採用する方針の選択肢
*   **短命 JWT とリフレッシュトークン:** JWT アクセストークンの有効期限を短く設定し、リフレッシュトークンローテーションと組み合わせて使用する。これにより、デナリストの必要性を最小限に抑える。
*   **Opaque トークンとの併用:** 機密性の高いリソースへのアクセスには Opaque トークンを使用し、即時無効化の恩恵を受ける。または、JWT アクセストークンを使用する場合でも、リソースサーバーがイントロスペクションエンドポイントを呼び出して有効性を確認することを必須とする。
*   **デナリストの限定的な使用:** 特定のセキュリティイベント（例：ユーザーのパスワード変更、アカウントロック）が発生した場合にのみ、デナリストを使用して JWT を即時無効化することを検討する。この場合、デナリストは高速な KVS（Key-Value Store）で実装する。
*   **送信者制約付きトークンの検討:** 将来的な拡張として、mTLS などの送信者制約付きトークンをサポートすることを検討する。これは OAuth 2.1 で推奨されるセキュリティ強化策である。

### 7. RFC 7662 `active=false` の基準: 期限切れ、無効化、未発行、誤ったオーディエンス/クライアント、`nbf` 前など。イントロスペクションレスポンスに表示されるべきクレーム。

#### 仕様の規定
RFC 7662 Section 2.2 にて、イントロスペクションレスポンスには、トークンが現在アクティブであるかどうかを示すブール値の `active` クレームが `MUST` 含まれる。`active=false` となる基準は以下の通りである: トークンが期限切れ (`exp`)、トークンがまだ有効期間前 (`nbf`)、トークンが無効化されている（例：無効化エンドポイント経由）、トークンが発行されたことがない、リクエスト元のクライアントがそのトークンの対象オーディエンス (`aud`) またはクライアント (`client_id`) と一致しない、トークンが不正な形式である。`active=true` の場合、以下のクレームが `SHOULD` 含まれる: `scope`, `client_id`, `username`, `token_type`, `exp`, `iat`, `nbf`, `sub`, `aud`, `iss`, `jti`。

#### ベストプラクティス
`active` クレームは常に含め、トークンの状態を明確に伝える。`active=false` となるすべての条件を正確に実装し、セキュリティ上の脆弱性がないようにする。特に、トークンが発行されたクライアントや対象オーディエンスとの不一致は、重要なセキュリティチェックである。`active=true` の場合、RFC 7662 で推奨されるすべてのクレーム（`scope`, `client_id`, `exp`, `iat`, `sub` など）を含めることで、リソースサーバーが必要な情報を取得できるようにする。機密性の高い情報（例：ユーザー名）は、リソースサーバーがその情報を取得する権限を持っている場合にのみ含める。

#### 採用する方針の選択肢
*   **RFC 7662 の厳格な実装:** `active` クレームを必須とし、`active=false` となるすべての条件（期限切れ、無効化、未発行、`nbf` 前、`aud`/`client_id` 不一致など）を正確に実装する。
*   **必須クレームの提供:** `active=true` の場合、`scope`, `client_id`, `exp`, `iat`, `sub`, `aud`, `iss`, `jti` などの主要なクレームを常に提供する。これにより、リソースサーバーはトークンに関する十分な情報を得られる。
*   **追加クレームの検討:** OIDC プロバイダとして、`acr`, `amr`, `auth_time` など、OIDC 固有のクレームをイントロスペクションレスポンスに含めることを検討する。これは、リソースサーバーがより詳細な認証コンテキストを必要とする場合に有用である。

### 8. OAuth 2.1 (draft-ietf-oauth-v2-1) および Security BCP (draft-ietf-oauth-security-topics) が無効化 / イントロスペクションに与える追加要件 (例: 送信者制約付きトークン、MTLS、リフレッシュトークンローテーションの相互作用)

#### 仕様の規定
OAuth 2.1 (draft-ietf-oauth-v2-1) Section 4.3.1 は、リフレッシュトークンが使用されるたびに新しいリフレッシュトークンを発行し、古いリフレッシュトークンを無効化することを `MUST` 要求する。また、認可コードが一度使用された後、再度使用された場合、その認可コードに関連付けられたすべてのトークン（アクセストークン、リフレッシュトークン）を無効化することを `MUST` 要求する。IETF Security BCP (draft-ietf-oauth-security-topics Section 2.2.2, 4.9) は、送信者制約付きトークン（例：mTLS クライアント証明書バインディング、DPoP）の使用を推奨し、トークンが盗まれた場合でも攻撃者がそれを使用することを困難にする。mTLS はクライアント認証およびトークンバインディングの強力なメカニズムとして推奨される。リフレッシュトークンは機密情報であり、厳重に保護されるべきであると強調しており、ローテーションはその主要な保護メカニズムの一つである。

#### ベストプラクティス
OAuth 2.1 で必須化されたリフレッシュトークンローテーションと認可コード再利用時のトークン無効化を厳格に実装する。これにより、リフレッシュトークンと認可コードの侵害リスクを大幅に低減できる。送信者制約付きトークン（特に mTLS）の導入を検討し、トークンの盗難に対する耐性を向上させる。これは、特に機密性の高いアプリケーションや環境において重要である。リフレッシュトークンは、認可サーバーの永続ストアに安全に保存し、ハッシュ化または暗号化して保護する。

#### 採用する方針の選択肢
*   **OAuth 2.1 の必須要件の完全実装:** リフレッシュトークンローテーションと認可コード再利用時のトークン無効化を完全に実装する。このプロジェクトは既に認可コード再利用時の兄弟トークン無効化を `grantId` で実現しているため、これをリフレッシュトークンローテーションにも適用する。
*   **送信者制約付きトークンの段階的導入:** まずは Opaque トークンと JWT トークンの両方で基本的な無効化・イントロスペクションを実装し、その後、mTLS や DPoP などの送信者制約付きトークンを段階的に導入することを検討する。
*   **リフレッシュトークンローテーションと無効化の連携:** リフレッシュトークンローテーション時に古いリフレッシュトークンを即座に無効化するだけでなく、その古いリフレッシュトークンに関連付けられていたアクセストークンも無効化する（または有効期限を短縮する）ことを検討する。

### 9. 主要 IdP (Auth0, Keycloak, Okta, Google) の実装例

#### 仕様の規定
主要な IdP は、OAuth 2.0 および OIDC の仕様に準拠してこれらのエンドポイントを実装している。
*   **Auth0:** RFC 7009 に準拠した無効化エンドポイントを提供し、リフレッシュトークン無効化時に関連するアクセストークンも無効化する。RFC 7662 に準拠したイントロスペクションエンドポイントを提供し、`active` クレームと標準クレームを返す。
*   **Keycloak:** RFC 7009 に準拠した無効化エンドポイントを提供し、リフレッシュトークン無効化時に、そのリフレッシュトークンから発行されたすべてのアクセストークンを無効化する。RFC 7662 に準拠したイントロスペクションエンドポイントを提供し、JWT アクセストークンの場合でも有効性を確認できる。
*   **Okta:** RFC 7009 に準拠した無効化エンドポイントを提供し、リフレッシュトークン無効化時に、関連するアクセストークンも無効化する。RFC 7662 に準拠したイントロスペクションエンドポイントを提供し、`active` クレームと標準クレームを返す。
*   **Google (OAuth 2.0):** RFC 7009 および RFC 7662 に準拠した無効化およびイントロスペクションエンドポイントを提供している。

#### ベストプラクティス
主要な IdP は、RFC 7009 と RFC 7662 のコア要件を厳格に実装しており、これは相互運用性とセキュリティの観点から重要である。リフレッシュトークン無効化時の関連アクセストークン無効化は、ほとんどの IdP で採用されている共通のセキュリティプラクティスである。JWT アクセストークンを使用している場合でも、イントロスペクションエンドポイントを提供することで、リソースサーバーがトークンの有効性を確実に検証できるようにしている。クライアント認証は、両エンドポイントで必須とされている。

#### 採用する方針の選択肢
*   **主要 IdP の挙動に倣う:** RFC 7009 と RFC 7662 の必須要件を完全に実装し、主要な IdP が採用しているセキュリティプラクティス（例：リフレッシュトークン無効化時の関連アクセストークン無効化）を取り入れる。
*   **JWT と Opaque トークンの両方に対応:** JWT アクセストークンと Opaque アクセストークンの両方に対して、無効化とイントロスペクションが正しく機能するように実装する。JWT の場合は、デナリストまたはイントロスペクションの強制を検討する。
*   **OIDC 固有の拡張:** OIDC プロバイダとして、イントロスペクションレスポンスに OIDC 固有のクレーム（例：`acr`, `amr`）を含めることを検討し、よりリッチな情報を提供する。

### このプロジェクトで採用すべき方針（推奨）

この TypeScript OIDC プロバイダライブラリは、JWT および Opaque アクセストークン、リフレッシュトークンローテーション、認可コード再利用時の兄弟トークン無効化、そして機密クライアント（`client_secret_basic` / `client_secret_post`）をサポートすることを前提とする。

1.  **無効化エンドポイント (RFC 7009):**
    *   `POST` メソッド、`token` パラメータ必須、`token_type_hint` 任意で実装する。
    *   機密クライアントには `client_secret_basic` または `client_secret_post` による認証を必須とする。
    *   トークンが存在しない場合でも `200 OK` を返すことで、サイドチャネル攻撃を防ぐ。
    *   レスポンスには `Cache-Control: no-store` および `Pragma: no-cache` ヘッダを必須とする。
    *   リフレッシュトークンが無効化された場合、その `grantId` に関連付けられたすべてのアクセストークン（Opaque および JWT）を無効化する。JWT アクセストークンについては、デナリストに JTI を登録することで即時無効化を試みる。
    *   アクセストークンが無効化された場合、デフォルトでは関連するリフレッシュトークンは無効化しない。ただし、クライアントからの明示的な要求（例：追加パラメータ）があった場合は、その `grantId` に関連付けられたリフレッシュトークンも無効化するオプションを提供する。

2.  **イントロスペクションエンドポイント (RFC 7662):**
    *   `active` クレームを必須とし、RFC 7662 Section 2.2 に記載されているすべての `active=false` 基準（期限切れ、無効化、未発行、`nbf` 前、`aud`/`client_id` 不一致、デナリスト登録など）を正確に実装する。
    *   `active=true` の場合、`scope`, `client_id`, `exp`, `iat`, `sub`, `aud`, `iss`, `jti` などの主要なクレームを常に提供する。
    *   OIDC プロバイダとして、`acr`, `amr`, `auth_time` など、OIDC 固有のクレームもイントロスペクションレスポンスに含めることを検討する。

3.  **OAuth 2.1 および Security BCP への対応:**
    *   OAuth 2.1 で必須化されたリフレッシュトークンローテーションを完全に実装し、古いリフレッシュトークンは即座に無効化する。
    *   認可コード再利用時の兄弟トークン無効化は、既存の `grantId` メカニズムを継続して利用する。
    *   JWT アクセストークンの即時無効化のために、デナリストを導入する。デナリストは高速な KVS で実装し、スケーラビリティを考慮する。
    *   将来的な拡張として、mTLS などの送信者制約付きトークンのサポートを検討し、トークンの盗難に対する耐性を向上させる。

これらの指針により、本ライブラリは最新の OAuth/OIDC 仕様に準拠し、高いセキュリティと相互運用性を提供できると考える。
