## 1. 目的・背景

### 1.1 Why(背景・課題)

自作の OpenID Provider(OP)ライブラリ(`@maronn-oidc/core` / `@maronn-oidc/cli`)を開発しているが、現状このライブラリの動作を第三者が試すには、CLI(`maronn-oidc`)でアプリケーションコードを生成し、依存関係のインストール・ビルドを行ったうえで自分の実行環境(Node.js サーバー等)にデプロイする必要がある。つまり「動くところを見る」までの初期コストが高く、以下の問題がある。

- ライブラリの利用検討者・記事読者が、コードを書かずに挙動を確認する手段がない
- 作者自身も、構成違い(クライアント設定・フロー差分)の OP を並べて検証するたびにセットアップ作業が発生する
- RP(クライアント)実装のテスト相手として使える「使い捨ての OP」が存在しない

### 1.2 What(目的 = 達成状態)

**Web UI 上で構成オプションを選択するだけで、検証用の OpenID Provider インスタンスが専用サブドメイン上に発行され、即座に OIDC フローを試せる状態**を実現する。

達成の判定条件: 利用者がログイン不要で、UI 操作のみで `https://{ランダム文字列}.{固定ドメイン}` の issuer と対応するクライアントクレデンシャルを入手し、Authorization Code フローを完走できること。

### 1.3 Benefit(便益)

| 対象者 | 変化 |
|---|---|
| ライブラリ利用検討者 | デプロイ不要で挙動を確認でき、採用判断が速くなる |
| RP 開発者 | テスト用 OP を数十秒で入手でき、RP 実装の検証が容易になる |
| 作者(Maronn) | ライブラリのショーケースとなり、記事・書籍・登壇からの導線として信用資産化する。構成差分の検証コストが下がり、CLI 生成コードの継続的な動作検証(ドッグフーディング)を兼ねる |

### 1.4 成功基準

- 初期リリース時点: OP 作成〜Authorization Code フロー完走まで、外部利用者がドキュメントを見ながら 5 分以内に到達できる
- OP インスタンスが TTL(24 時間)経過後に確実に削除され、放置インスタンスが残らない
- 【要確認】公開後の定量目標(作成された OP 数/月 など)は未設定。設定するか要判断

### 1.5 非目的(やらないこと)

- 本番用途の OP ホスティング(あくまで検証用。SLA なし、データは 24 時間で消滅)
- 利用者アカウント管理・ログイン機能(abuse 対策はレート制限で代替)
- 課金・商用サービス化(初期リリース時点では対象外)

## 2. 用語

| 用語 | 意味 |
|---|---|
| OP | OpenID Provider。本アプリが生成する検証用 IdP インスタンス |
| OP インスタンス | 1 回の作成操作で発行される、サブドメイン単位の独立した OP |
| Dispatcher | ワイルドカードサブドメイン宛の全リクエストを受け、対象 OP インスタンスへ振り分ける受付レイヤ |
| コアライブラリ | `@maronn-oidc/core`(0.x 系)。**フレームワーク非依存のロジック層**で、production dependencies ゼロ・Web 標準 API(Fetch API / Web Crypto API 等)のみで動作する。Hono に依存しているのはコアではなく CLI の hono ターゲット生成コード |
| CLI | `@maronn-oidc/cli`(コマンド名 `maronn-oidc`)。hono / express / fastify / nextjs の 4 フレームワーク向けに、ルーティング・ストア・画面込みの動作可能な OP コードを生成する。現行(v0.0.1)の生成内容はフレームワークごとに固定だが、**将来的にはオプションで生成パターン(機能構成)を変えられるようにする構想**であり、Playground はその受け皿となる。Playground は **hono ターゲット**を使用する(Workers 上で動作させる自然な選択肢のため) |
| 生成コード | CLI が出力する TypeScript ソース一式(`app.ts` / `apply.ts` / `config.ts` / `store.ts` / `resolvers.ts` / `views.ts` / `routes/*` / `conformance.test.ts`)。`hono` と `@maronn-oidc/core` を import するため、実行にはインストールとビルド(esbuild / tsc)が必要 |
| テストユーザー | OP インスタンス内にのみ存在するダミーユーザー(最大 5 名) |

## 3. スコープ

### 3.1 初期リリース(v1)

- OIDC Basic OP プロファイル相当の機能(Authorization Code フロー)。CLI 生成 OP が実際に備えるエンドポイントは以下の通りで、Basic OP 必須分に加えて refresh token / introspection / revocation も最初から含まれる
  - `/.well-known/openid-configuration`(discovery)、`/.well-known/jwks.json`
  - `/authorize`(GET / POST)、`/token`(authorization_code / refresh_token)、`/userinfo`(GET / POST)
  - `/introspect`(RFC 7662)、`/revoke`(RFC 7009)
  - `/login`、`/consent`(OP 内蔵のログイン・同意画面)
  - PKCE(S256)必須(OAuth 2.1 方針。conformance 互換フラグ `allowNonPkceAuthorizationCodeFlow` あり)、ID Token 署名 RS256(ES256 も選択可)、`prompt`(none / login / consent / select_account)、Request Object(署名付き)、`claims` パラメータ、RFC 9207 `iss` レスポンスパラメータ、CORS 対応
- Static クライアント(作成時に client_id / client_secret を発行)。クライアント定義は生成コードの `RegisteredClient` 型(clientType confidential / public、grantTypes、tokenEndpointAuthMethod client_secret_basic / client_secret_post、offlineAccessAllowed 等)に従う
- 固定の**ユーザー名+パスワード**によるログイン画面(生成コードのログインフォームは `username` / `password` フィールド。メールアドレスでログインさせたい場合は、テストユーザー定義の username に email 文字列を登録する運用とするか、`views` 差し替えでラベルを変更する)
- テストユーザーの事前定義(最大 5 名)
- TTL 24 時間での自動削除
- OP 作成 API・生成された OP エンドポイント双方へのレート制限
  - 補足: ログイン画面のブルートフォース対策(試行回数制限、上限超過で 429)はコアライブラリの `handleLoginFailure` として生成コードに組み込み済み

### 3.2 将来拡張(ロードマップ、v1 では実装しない)

コアライブラリは機能をコアモジュールの組み合わせで構成し、CLI が設定済みのアプリコードを生成する設計のため、ライブラリ側に機能が追加されるたびに Playground の選択肢として露出させる。想定される拡張(リポジトリの tasks / study-material で検討中のものを含む):

- Dynamic Client Registration(現状のコアは `registration` パラメータを明示拒否しており未実装)
- DPoP(T-019 として検討中)、CIBA、PAR、JAR 等の拡張仕様
- 認証方式の拡張(パスキー/WebAuthn 等)
- response_type / client 認証方式 / PKCE 要否 / アクセストークン形式(jwt / opaque は ProviderConfig で既に選択可能)などの詳細な構成オプション
- **CLI の構成オプション対応**: CLI にオプションを渡すことで生成される機能構成(有効化するエンドポイント・フロー・拡張仕様)そのものを変えられるようにする。これが実現した段階で、Playground の作成 API は「UI の選択オプション → CLI オプション変換 → コード生成」という本来目指す形へ移行する(§4 / §6.2 参照)
- hono 以外の生成ターゲット(express / fastify / nextjs)の挙動差分を試す構成

## 4. ユーザーフロー

1. 利用者が Playground の Web UI にアクセスする(ログイン不要)
2. 構成オプションを選択する
   - v1 の選択項目: 【要確認: v1 で UI に出す選択肢の確定。最小案は「テストユーザー定義のみ」で、フロー等は固定】
   - テストユーザー(username / password / 表示名)を最大 5 名まで定義する
3. 「OP を作成」を実行する
   - **目指す形**: システムは UI の選択オプションを CLI のオプションに変換してコード生成を実行し、生成物をデプロイする。ただし現行 CLI(v0.0.1)のオプションは `generate|setup <framework> --output <dir> [--entry <file>]` のみで、テストユーザー・クライアント・フロー等の構成オプションはまだ受け付けない(将来対応する構想。§3.2 参照)
   - **v1 の実現方式**: CLI がオプション対応するまでは、「UI の選択オプション」を**生成コードの注入ポイント**に対して設定として与える。注入ポイントは生成コードに既に存在する:
     - `applyOidc(app, options)` の `ApplyOidcOptions`: `config`(issuer / 各種 TTL / accessTokenFormat 等)、`signingKeyProvider`(必須)、`clientResolver` / `tokenClientResolver`、`sessionResolver` / `consentResolver`、`acrResolver`、`jwksProvider`、`corsOrigins`、`views`
     - `samples/hono-cloudflare/src/app.ts` がCloudflareバインディング(`ISSUER`、`OIDC_CLIENTS_JSON`、D1等)から設定を読んで `applyOidc` に渡す実例であり、Playground のエントリファイルはこれと同型になる
   - システムは生成コード+インスタンス設定から動作可能な Worker スクリプトを組み立て、Cloudflare Workers へデプロイする(6.2 参照)
   - デプロイには数秒〜数十秒かかるため、UI は進行状況を表示する非同期処理とする
4. デプロイ完了後、システムが以下を発行し、UI に表示する
   - issuer URL: `https://{ランダム文字列}.{固定ドメイン}`
   - client_id / client_secret(Static クライアント)
   - redirect_uri の登録内容 【要確認: 利用者が入力するのか、複数登録を許すか。なおコアの redirect_uri 検証は**完全一致**のみで(fragment 付き・危険スキームは拒否)、ワイルドカード緩和はライブラリが対応していないため選択肢にならない】
   - 有効期限(作成から 24 時間)
5. 利用者は自分の RP から当該 issuer に対して Authorization Code フローを実行する
   - ログイン画面では手順 2 で定義したテストユーザーの資格情報を入力する
6. 24 時間経過後、OP インスタンスは自動削除される。以降のリクエストにはエラーを返す 【要確認: 404 か、期限切れである旨の専用エラーか】

## 5. 機能要件

### 5.1 OP 作成

- 作成操作 1 回につき OP インスタンスを 1 つ発行する
- サブドメインはランダム文字列(推測困難・衝突チェックあり)【要確認: 文字種と長さ】
- 作成時に以下を生成する
  - インスタンス固有の署名鍵ペア(インスタンス間で共有しない)。**鍵生成はライブラリの責務外**(生成コードは `SigningKeyProvider` の実装を要求するだけで鍵を作らない)ため、Playground 側が Web Crypto API で RSA 2048(RS256)鍵ペアを生成し、`SigningKeyProvider` 実装経由でインスタンスに与える。コアの `createCachedSigningKeyProvider` / `assertKeyStrength` 等のヘルパーが利用できる
  - Static クライアント 1 件(client_id / client_secret)。生成コードの `RegisteredClient` 形式で定義し、`createInMemoryClientResolver(clients)` に渡す(`samples/hono-cloudflare` の `OIDC_CLIENTS_JSON` 方式が転用できる)【要確認: 複数クライアントの同時発行を許すか】
  - テストユーザー(利用者定義、最大 5 名)
    - **実装事実**: 生成コードは `ProviderStores` / `JsonStoreBackend` を `applyOidc` の `storage` へ注入でき、ログインとUserInfoも同じ永続ユーザーストアを参照する。Playgroundはユーザー定義をD1へ投入する初期化処理を実装すればよく、生成後の `store.ts` 書き換えは不要

### 5.2 OP インスタンスの挙動

- コアライブラリ+CLI 生成コードの Basic OP 相当機能を提供する(エンドポイント一覧は 3.1 参照)
- 認証は**ユーザー名+パスワード**の固定フォームのみ(ログイン試行回数制限あり)
- テストユーザーはインスタンス内でのみ有効。インスタンス間でデータを共有しない
- 生成コードの `conformance.test.ts` が担保する Basic OP 挙動を基準とする。Playground がストアやユーザー定義を差し替える場合も、この契約テストが通る範囲のカスタマイズに留める

### 5.3 ライフサイクル(TTL)

- 有効期間は作成から 24 時間固定。延長・手動削除は提供しない
- 期限切れインスタンスはバッチ的に削除する(署名鍵・クライアント・ユーザー・発行済みトークン状態を含む全データ)
- 削除の実装方式は 6.3 参照

### 5.4 Abuse 対策

- **OP 作成 API**: IP ベースのレート制限を掛ける 【要確認: 上限値。例: 同一 IP から 10 件/時】
- **生成された OP のエンドポイント**: 前段に共通の受付レイヤを置かない構成のため、①ゾーンのレート制限ルール(`*.{固定ドメイン}/*` を対象、無料枠のルールで対応)②生成コードに組み込むレート制限ミドルウェア、のいずれかまたは併用で実現する 【要確認: 実装方式と上限値。なおライブラリ/生成コードはエンドポイント汎用のレート制限ミドルウェアを提供していない(ログイン試行制限のみ組み込み。検討資料: `study-material/rate-limiting-and-brute-force.md`)ため、v1 は①ゾーンルールを第一候補とするのが実体に合う】
- 秘密情報(署名鍵・client_secret)はコード埋め込みではなく Secrets / バインディング経由で投入することを原則とする。ただし本アプリの主目的は検証用 OP の提供であり、Secrets 対応のために生成・デプロイ工程が過度に複雑化する場合は、簡素さを優先してよい
- 追加検討: 作成時の Turnstile(CAPTCHA)挿入 【要確認: v1 で入れるか】

## 6. アーキテクチャ

### 6.1 全体構成

Cloudflare Workers 上に構築する。OP 作成のたびに生成コードベースの Worker スクリプトを組み立て、インスタンスごとに独立した Worker としてデプロイする方式を取る。固定ドメインにはワイルドカード DNS(`*.{固定ドメイン}`)を設定し、サブドメイン宛リクエストを対象インスタンスの Worker に届ける。

**実装済み基盤(2026-07-23)**: `samples/hono-cloudflare` はCLIのhonoターゲット生成コードをCloudflare Workers上で実行し、全OP状態をD1へ保存する。Wrangler/workerdとローカルD1を使うPlaywright E2EでBasic OPの主要フローを完走済み。Playgroundではこの構成を基準に、インスタンスごとのD1分離・鍵管理・自動削除を追加する。

```
[作成時]
管理 UI --(選択オプション)--> 作成 API
                                  |
                                  v
                       生成コード + インスタンス設定の合成
                       (コアライブラリを含む
                        動作可能なアプリコード)
                                  |
                                  v
                       Cloudflare Workers へデプロイ
                       (サブドメイン = インスタンス名)
                                  |
                                  v
                       issuer URL / client_id /
                       client_secret を UI に返却

[利用時]
利用者の RP --> https://{sub}.{固定ドメイン} --> 当該インスタンスの Worker
```

- **Workers for Platforms は採用しない**(Workers Paid とは別に月額 $25 の専用プランが必要で、コスト制約に反するため)。通常の Workers として、インスタンスごとに個別の Worker スクリプトをデプロイする
  - ルーティング: ゾーンにワイルドカード DNS(`*.{固定ドメイン}`、proxied)を設定し、デプロイ時に Cloudflare API で `{sub}.{固定ドメイン}/*` の Workers Route を当該スクリプトに紐付ける。削除時は Route とスクリプトを両方削除する
  - 制約: Workers Paid のアカウントあたりスクリプト数上限(500 前後)が同時稼働インスタンス数の上限になる。TTL 24 時間+作成レート制限があるため v1 の規模では十分と判断する 【要確認: 実装前に現行の上限値を公式ドキュメントで再確認】
- デプロイオーケストレータは 2 案を段階的に検討する(下記 6.2.1)
- Playground の管理 UI(OP 作成画面)は専用ホスト(例: apex または `app.{固定ドメイン}`)で提供する 【要確認: UI のホスト名】

### 6.2 コアライブラリ・CLI との関係

本アプリの中核は「実際のユーザーが CLI で得るものと同一の生成コードを、そのままデプロイして動かす」ことにある。

- コアライブラリ(0.x 系)はフレームワーク非依存のロジック(validate / issue / resolver・store 契約)を提供し、CLI がフレームワークごとにルーティング・画面・ストア込みの動作可能なアプリケーションコードを生成する。Playground は hono ターゲットを使う
- **設定の与え方(二段構え)**:
  - **目指す形**: 作成 API が UI の選択オプションを CLI のオプションに変換してコード生成を実行する。CLI のオプションによる生成パターン変更は現行(v0.0.1)では未実装だが将来対応する構想であり(§3.2)、対応後は生成される機能構成そのものを UI から選べるようになる
  - **v1(CLI オプション対応まで)**: Playground 固有の可変部(issuer・クライアント・テストユーザー・署名鍵・redirect_uri)は、生成コードの注入ポイント(`applyOidc` のオプション / Hono context / 環境変数・Secrets)に与える。エントリファイル(app 相当)だけが Playground 固有で、生成コード本体は利用者が CLI で得るものと同一に保つ — samples が採っている構成と同じであり、ドッグフーディングの純度はこの形で担保する。設定データ系の可変部(ユーザー・クライアント・鍵)は CLI オプション対応後も注入ポイント経由が適する可能性が高く、CLI オプションは機能構成の切り替えを担う住み分けを想定する
- この構成により、Playground はライブラリだけでなく **CLI の生成コードそのものの継続的な動作検証(ドッグフーディング)** を兼ねる。ライブラリ・CLI に機能が追加されれば、UI に選択肢を追加するだけで Playground でも試せるようになる
- 生成コードは `JsonStoreBackend` / `ProviderStores` の注入契約を持ち、未指定時のみ契約テスト用インメモリ実装へフォールバックする。`samples/hono-cloudflare` はD1アダプターを注入する。詳細は6.3参照

#### 6.2.1 デプロイオーケストレータ(コード生成〜デプロイの実行環境)

制約: 無料の範囲、または Workers Paid 契約に含まれる機能のみで構成する。

**調査結果(2026-07-05、実装事実)**: 旧案 A の前提「生成が『事前バンドル済みのライブラリコード+設定モジュールの合成』で表現できること」は、**現行 CLI の出力そのものでは満たされない**。CLI は TypeScript ソースを出力し、生成コードは npm パッケージとして `hono` と `@maronn-oidc/core` を import するため、実行可能にするには依存インストールとバンドル(samples では esbuild)が必要であり、この工程は Worker 内で実行できない。一方で「v1 の可変部が設定データ(テストユーザー・クライアント・redirect_uri・鍵)のみ」という見立ては生成コードの注入ポイント設計(6.2)と整合しており正しい。これを踏まえ案を再定義する。

**案 A'(目標形): 事前バンドル+設定合成(作成 API Worker 内で完結)**

CI(またはリリース工程)で「CLI で hono ターゲットを生成 → Playground 用エントリファイルと合わせて esbuild で単一 JS(ESM)にバンドル」した**共通テンプレートスクリプト**を成果物として作成 API Worker に同梱する。作成 API は作成リクエストごとに、テンプレートスクリプト+インスタンス設定(テストユーザー・クライアント・issuer)を Cloudflare API でアップロードし、署名鍵・client_secret は Secrets として設定し、Route を作成する。

- 利点: 追加インフラ不要、デプロイ完了まで数秒、シークレットが Worker 内で完結
- 前提条件: バンドルは作成時ではなく事前に 1 回だけ行う。設定はコードに埋め込まず、環境変数・Secrets・D1バインディング経由で注入する(`samples/hono-cloudflare` の方式)
- ドッグフーディングとの関係: バンドル対象の生成コードは CLI 出力そのもの。CLI・ライブラリの更新時はテンプレートスクリプトを再バンドルして差し替える(この再バンドルを CI に組み込めば、生成コードの Workers 動作検証が継続的に回る)
- 将来 CLI がオプションによる生成パターン変更に対応した場合: UI に露出するオプションの組み合わせごとに事前バンドル済みテンプレートを CI で複数用意する(組み合わせが少数のうちは成立)。組み合わせが増えて事前バンドルが破綻する段階では、作成時に CLI を実行できる案 B(またはビルド可能な別基盤)への移行を再検討する

**案 B(フォールバック): GitHub Actions 経由**

作成 API が `repository_dispatch` で GitHub Actions ワークフローを起動し、Actions 上で CLI をそのまま実行(生成→インストール→esbuild バンドル→ `wrangler deploy`)する。

- 利点: CLI と生成〜ビルド工程を一切改造せずそのまま使える(ドッグフーディング純度が最も高い)。作成のたびに最新の CLI 出力を使える
- 欠点: レイテンシ(ランナー起動含め 30 秒〜2 分)、シークレットの受け渡し設計が必要(client_secret・署名鍵は Actions 側で生成し、結果を作成 API にコールバックする形が安全)、非公開リポジトリの場合は無料枠 2,000 分/月の消費
- 採用条件: 案 A' のテンプレートスクリプト方式で不都合が出た場合。特に、CLI のオプション対応後に UI 選択肢の組み合わせが増え、事前バンドル済みテンプレートの網羅が現実的でなくなった場合の受け皿となる(作成ごとに CLI をオプション付きで実行できるのは案 B のみ)

**方針**: v1 の可変部は設定データのみで案 A' の前提を満たすため、**案 A' を目標形**とする。まず「CLI 生成 → esbuild バンドル → workerd で Basic OP フロー完走」を手元で成立させ(6.1 の前提タスク)、成立しない障害があれば案 B で出して将来 A' へ移行する。

### 6.3 状態管理と削除

状態は 2 層に分かれる。

**(a) Playground 管理側の状態**: どのサブドメインに、いつ、どの構成でデプロイしたか(TTL 期限、スクリプト名、プロビジョニングしたリソースの識別子)。KV または D1 で保持し、Cron Triggers で期限切れインスタンスを走査して削除処理を実行する。

**(b) OP インスタンス内部の状態(実装事実)**: 生成コードは auth transaction / authorization code / access token / refresh token / auth session / browser session / consent / user を一つの `JsonStoreBackend` 上へ構成できる。`samples/hono-cloudflare` はD1アダプターを実装済みで、Wranglerローカル実行でも状態が永続化される。PlaygroundもD1を第一候補とし、インスタンスごとにDBを分けるか共有DB内でnamespaceを分けるかを決定する必要がある。

- authorization code再利用検知とrefresh token rotationはusedフラグを残すため、Playgroundでも生成済み`ProviderStores`をそのまま使い、個別resolverで物理削除へ置き換えない
- KVは結果整合性がauthorization codeの即時交換と相性が悪いため、主要状態の保存先には採用しない

**設定・秘密情報の持ち方**: テストユーザーとOP状態はD1、クライアント定義・issuerは環境変数、署名鍵(秘密鍵)・client_secretはSecretsバインディングで注入し、エントリファイルがそれらを読んで`applyOidc`へ渡す(`samples/hono-cloudflare`を基準に鍵管理を強化する)。

**削除処理**(TTL 24 時間経過後、Cron でバッチ実行):
1. Worker スクリプトと Workers Route の削除
2. インスタンス用にプロビジョニングしたバインディングリソース(Durable Object / KV namespace 等、6.3(b) の決定に従う)の削除
3. 管理側メタデータの削除

### 6.4 鍵管理

- 署名鍵はインスタンス作成時に個別生成し、インスタンス状態として保持する(生成主体は Playground。ライブラリは `SigningKeyProvider` 契約で鍵を受け取る側)
- v1 は RS256(RSA 2048)を必須の既定とする。コアは ES256 等の複数鍵登録・JWKS への広告(`getSigningKeys`)にも対応しているため、署名アルゴリズム選択は将来の UI オプション候補
- 全インスタンス共通の鍵は使用しない(あるインスタンスの鍵漏洩・悪用が他に波及しない)
- インスタンス削除時に鍵も破棄する

## 7. 非機能要件

| 項目 | 内容 |
|---|---|
| 可用性 | 検証用途のため SLA なし。ベストエフォート |
| データ保持 | 全データは作成から 24 時間で消滅。個人情報は扱わない前提(テストユーザーはダミー) |
| コスト | Workers Paid($5/月)契約の範囲内+無料枠で構成する。有料アドオン(Workers for Platforms 等)や有料の外部 SaaS は使用しない。レート制限と TTL でリソース消費を抑制する |
| 可観測性 | 作成数・アクティブインスタンス数・レート制限発動数を計測できること 【要確認: 手段(Workers Analytics 等)】 |
| セキュリティ | 検証用である旨を UI・discovery 周辺で明示する。生成された OP を本番利用しないよう注意書きを表示する。生成コードのセッション Cookie は `HttpOnly; Secure; SameSite=Lax` 固定のため HTTPS 配信が前提(Workers + proxied DNS で満たされる) |
