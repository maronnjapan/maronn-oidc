# v0.x リリーススコープ（確定）

> これは pX タスクではなく**リリース判定の確定文書**。v0.x を出してよいかは本書のチェックリストで機械的に判定する。Codex セカンドオピニオン（2026-05-16）を反映した確定版。

## ステータス

🟢 確定 / 2026-05-16

## 前提（戦略フレーム）

- このOSSは製品ではなく、**IDaaS導入支援コンサルへのファネル**。
- 初期ターゲット = **ID管理が全然できていない中小企業**（専任ID/インフラ担当なし、本番自前運用能力なし）。
- ファネルは「意図的に本番級にしない」では作らない。**責務の境界**で作る:
  - スコープ内: 仕様の検証・体感
  - スコープ外: 本番運用 / 鍵管理 / 監査 / 可用性 / 移行設計（＝コンサル着地点）
- 差別化軸「Speed=最新仕様最速」は初期SMEセグメントの価値とズレる。**Speedはブランド/将来拡張向けの長期投資**と位置づけ、v0.x の顧客価値は「ゼロから"何を買うべきか理解"までの最短路」とする。

## 成熟度Tier定義

| Tier | 定義 | 役割 |
|---|---|---|
| A: Demonstrable | **非専門家が**数分でハッピーパスをe2e体感でき、トークン/クレームを観察できる | v0.x の出荷ライン |
| B: Conformant | 該当Conformance Suiteを通過 | v1.0条件。ブランド/コンサル信用の中期保険（ファネル駆動力ではない） |
| C: Production-grade | エッジ・セキュリティ・運用完備 | **恒久的にスコープ外**（コンサル/IDaaS移行の領域） |

## リリース条件

- **v0.x 公開条件** = 下記チェックリストが全て満たされる
- **v1.0 条件** = Basic OP が Tier B（Conformance）達成。v0.x の near-term ゲートにはしない。

## v0.x リリース判定チェックリスト

### A. 検証シナリオ（全て Tier A ＝非専門家が体感・理解できる）

- [ ] Googleアカウントでログインできる（フェデレーション/ソーシャルログインの体感）
- [ ] 1ログインで複数サンプルアプリに入れる（SSOの体感）
- [ ] 追加認証を挟む＝再ログイン要求が効く（`prompt=login` / `max_age`）
- [ ] 同意画面が出る/出ないを切り替えられる（`prompt=consent`）
- [ ] メール・名前がアプリにどう渡るか可視化できる（claims / UserInfo）
- [ ] 「これを本番でやるには」→ IDaaS選定ガイドに着地する導線がある（**ファネルの心臓**）

> 上記はコア仕様（Authorization Code + PKCE + ID Token(RS256) + UserInfo + prompt）の見せ方をSME可読にしたもの。仕様実装の追加は不要。

### B. 非本番ガードレール（成果物として存在すること）

- [ ] `SECURITY.md`（脆弱性方針・サポート対象/対象外を明記）
- [ ] サポート対象外の明示（本番運用は非対象である旨をREADME等の入口に）
- [ ] 開発用途専用の安全なデフォルト（秘密鍵・シークレットの取り扱いがdev前提で安全側）
- [ ] 非対象範囲の明文化（永続化 / 鍵管理 / 監査 / HA はスコープ外）

### C. ファネル導線（リリース要件、仕様カバレッジ外）

- [ ] 「誰向けか」が入口で分かる（ID管理ゼロの中小企業向けと明示）
- [ ] 「何が試せるか」がシナリオ単位で提示されている
- [ ] 「どこから先は相談対象か」（責務の境界）が示されている
- [ ] 「どう移行するか／問い合わせ口」＝ IDaaS選定・移行ガイドへの動線

### D. Speedシグナル（最小コストで看板を守る）

- [ ] 次に何を追うかの**優先順位ロジック**を公開（4軸: 相談化しやすさ / 採用の近さ / 実装コスト / 移行接続性。＋ブランド面で「新しさ」）
- [ ] 後続ロードマップを公開
- [ ] （任意・ブロッカーにしない）拡張仕様を1本だけ先行Tier A

## v0.x に含めないもの（「いつまでも出せない」罠の回避）

- 先端仕様（PAR / DPoP / CIBA / Token Exchange / FAPI 2.0 / OID4VCI）
  - 初期SMEセグメントは一生相談しない。後続ロードマップの**駆動力にもしない**。
- Conformance取得（v1.0送り）
- 本番ハードニング（恒久スコープ外）

## 後続ロードマップ（先端仕様の代わり＝SME関連の深化）

優先順位は 4軸スコアで決める（新しさ単独で決めない）:

1. ソーシャル/フェデレーションのプロバイダ拡充（Google/Microsoft）
2. 複数サンプルアプリでのSSO体験
3. 基本的なステップアップ認証
4. ユーザー/同意管理のUX
5. IDaaS選定・移行ガイドのテンプレート化

## 未決事項（事業側判断・v0.xは非ブロッカー）

- go-to-market メッセージを「最速で最新仕様」ではなく「ID管理ゼロから最速で要件把握」に振るかの最終確定。Speed軸の対外的な位置づけは事業側の意思決定。
